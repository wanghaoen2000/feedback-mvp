## 1.1 项目背景

「学情反馈系统」（项目代号 feedback-mvp）是一个面向教育行业的智能教学辅助平台，核心目标是利用 AI 能力自动化生成课后学情反馈报告，同时提供作业管理、作业批改、批量文档处理等配套功能。

### 系统解决的核心痛点

- 教师每次课后需要花费大量时间手工撰写学情反馈，包括课堂表现总结、知识点掌握分析、复习建议等。
- 作业批改需要逐一处理文字、图片等多模态输入，效率低下。
- 学生状态跟踪分散在各处，缺乏系统化管理。

### 目标用户

- **一对一辅导教师**（主要场景）
- **小班课教师**（3~10 人班级）
- **教学管理者**（查看系统运行状态、管理用户权限）

### 核心价值

- 一键生成课后学情反馈（Markdown 格式），自动上传至 Google Drive
- 同时并行生成复习文档、测试本（Word 格式）和气泡图（PNG 格式）
- 支持后台执行 — 提交任务后可关闭浏览器，任务在服务端继续完成
- 作业管理系统支持语音输入 → AI 结构化 → 学生状态迭代更新
- 作业批改系统支持文字 + 图片 + 文件多模态输入，AI 自动批改并回写学生状态
- 批量处理系统支持一次性处理数十到上百个文档生成任务

### 1.3.2 通信机制

系统使用两种前后端通信方式，各有其适用场景：

**1. tRPC（JSON-RPC over HTTP）**

- 用于所有非流式的请求-响应操作（增删改查、配置管理、任务提交等）
- 端到端类型安全：前端调用 `trpc.xxx.yyy()` 时，TypeScript 自动推断参数和返回值类型
- 统一的错误处理：所有错误通过 TRPCError 抛出，前端统一 catch
- 请求路径：`/api/trpc/*`

**2. SSE（Server-Sent Events）**

- 用于实时流式生成（学情反馈、复习文档、测试本等长时间 AI 生成任务）
- 单向推送：服务器持续向客户端发送文本片段，客户端拼接显示
- 事件类型：`start`（开始）→ `progress`（进度/文本块）→ `complete`（完成）或 `error`（失败）
- 支持客户端中断：前端关闭连接后，后端通过 AbortController 取消 AI 调用
- 请求路径：`/api/feedback-stream`、`/api/class-feedback-stream`、`/api/review-stream` 等

> **设计决策：为什么同时用 tRPC 和 SSE？**
>
> tRPC 不原生支持 Server-Sent Events 的流式推送。tRPC 的 subscription 机制基于 WebSocket，但本系统不需要双向通信，SSE 更轻量且无需额外的 WebSocket 服务。因此，普通操作走 tRPC，流式生成走原生 Express SSE 端点。

### 1.3.3 认证与权限

系统通过 Manus 平台的 OAuth 实现用户认证，登录后使用 Cookie-based Session 维持会话。

**用户角色：**

| 角色 | 权限范围 | 典型操作 |
|------|----------|----------|
| user（普通用户） | 只能操作自己的数据 | 生成反馈、管理自己的学生、批改作业 |
| admin（管理员） | 可管理所有用户 | 创建/暂停/删除用户、修改角色、伪装登录 |

**权限守卫（三级）：**

- `publicProcedure` — 不需要登录（仅用于 `auth.me` 和 `auth.logout`）
- `protectedProcedure` — 需要有效登录且账号状态为 active
- `adminProcedure` — 需要 admin 角色

**数据隔离机制：**

所有用户数据表都包含 `userId` 列，每次查询和写入都以当前登录用户的 `userId` 作为过滤条件。即使知道其他用户的数据 ID，也无法跨用户访问。管理员通过「伪装登录」（impersonate）功能切换身份来查看/调试其他用户的数据，而不是直接绕过隔离。

## 1.4 功能模块总览

系统的前端界面以顶部 Tab 页组织，共包含四大功能模块，外加全局设置和路书管理两个弹窗。

### 1.4.1 课堂反馈（Tab 1）

核心功能模块，支持一对一和小班课两种模式，完成从输入到输出的完整反馈生成流水线。

**生成产物（5 步流水线）：**

| 步骤 | 产物 | 格式 | 说明 |
|------|------|------|------|
| 1. 学情反馈 | 课后反馈报告 | Markdown → Google Drive | 主报告，包含课堂表现、知识点分析、建议等 |
| 2. 复习文档 | 复习资料 | DOCX → Google Drive | 根据反馈内容生成结构化复习材料 |
| 3. 测试本 | 测试题 | DOCX → Google Drive | 根据课程内容生成测试题和答案 |
| 4. 课后信息提取 | 结构化数据 | Markdown → Google Drive | 提取课程关键信息，可导入作业管理 |
| 5. 气泡图 | 可视化图表 | PNG → Google Drive | 学生各维度表现的气泡图 |

**输入材料：**

- **上次课反馈** — 可从 Google Drive 自动读取，也可手动粘贴/上传
- **课堂笔记（必填）** — 可从本地下载目录自动读取，也可手动粘贴/上传
- **录音转文字（必填）** — 可从本地下载目录自动读取，支持多段录音合并

**执行模式：**

- **实时模式（SSE）** — 页面保持打开，实时看到生成进度和文本
- **后台模式（Background Task）** — 提交后可关闭页面，通过「任务记录」查看状态

### 1.4.2 学生管理（Tab 2）

管理学生名册和作业状态，是教学管理的核心模块。

- **学生名册** — 添加/删除学生，设置日计划/周计划类型
- **语音输入处理** — 语音转文字 → AI 结构化 → 预入库队列 → 确认入库
- **学生状态管理** — 每个学生维护一个「当前状态」文档，每次入库迭代更新
- **一键打分** — 选择日期范围 + 打分要求，AI 为全部学生批量生成评分
- **打分同步** — 打分结果可一键同步回各学生的状态文档
- **作业提醒** — AI 根据学生状态生成个性化提醒文案
- **数据备份/恢复** — 导出/导入学生数据（Markdown 格式），支持 Google Drive 自动备份

### 1.4.3 作业批改（Tab 3）

支持多模态输入的 AI 自动批改系统。

**输入方式：**

- **文字输入** — 直接粘贴文本
- **图片输入** — 拖拽/粘贴/上传图片（自动压缩，支持多张）
- **文件输入** — 上传 Word (.docx)、PDF (.pdf)、文本 (.txt) 文件

**批改类型（可自定义）：**

- 豆包翻译 — 检查翻译准确性、流畅性、用词
- 学术文章 — 检查论点逻辑、论据、用语规范性
- 日常文章 — 检查语法、用词、表达地道性
- 词汇填空 — 检查答案正确性、解释错误选项

**输出：**

- **批改结果** — 详细的批改内容，一键复制
- **状态更新** — 自动推送到学生管理系统，更新学生的当前状态

### 1.4.4 批量处理（Tab 4）

面向需要一次性处理大量文档的场景，如批量生成教学材料。

- 支持指定任务编号范围（如 1~100）
- 可控并发数（默认 50，上限 200）
- 多种输出模板：教学材料（带样式）、通用文档、Markdown、词汇卡片、写作素材
- 支持独立文件（每个任务一个）和共享文件（所有任务共用）
- 实时进度追踪，单个子任务失败可独立重试

### 1.4.5 全局设置与路书管理

**全局设置（弹窗）：**

- API 配置 — 模型选择、API Key、API 地址、最大 Token 数
- API 供应商预设 — 预配置多个供应商，一键切换
- Google Drive 连接 — OAuth 授权/断开
- 存储路径 — Google Drive 基础路径、分类路径、本地路径
- 系统诊断 — 一键检测 API、Drive、数据库等各项连通性
- 管理员面板 — 用户管理（创建/暂停/删除/伪装/改角色）

**路书及范例管理（弹窗）：**

- 一对一路书 — 指导 AI 生成反馈的风格、格式、重点
- 小班课路书 — 小班课专用的 AI 指导模板
- 首次课范例 — 新生第一次课的反馈模板（一对一 + 小班课各一份）

## 1.6 版本管理与协作模式

### 1.6.1 版本号规则

系统使用简单递增版本号（V1、V2、……V179），不使用语义化版本。每次发布前手动更新 `scripts/generate-version.cjs` 中的 `VERSION` 常量。构建时自动生成 `version.generated.ts`，前端显示在页面右上角。

**当前版本：V179**

版本从 V1 迭代至今，经历了从最初的基础反馈生成到完整的教学辅助平台的演进。

### 1.6.2 协作模式

项目采用双角色协作开发模式：

| 角色 | 职责 | 工具 |
|------|------|------|
| Claude（开发端） | 功能开发、Bug 修复、推送 GitHub 分支 | Claude Code CLI |
| Manus（部署端） | 合并分支 → 构建 → 部署 → Checkpoint | Manus 沙箱环境 |

两端通过 `COLLAB.md` 文件进行异步沟通。Claude 在推送代码前将变更说明写入 COLLAB.md 的「Claude → Manus」区域，Manus 部署后在「Manus 反馈区」填写部署结果。

**部署流程（严格顺序）：**

1. `git fetch origin`
2. `git merge origin/claude/xxx`（应直接 fast-forward）
3. `npm install`（如有依赖变更）
4. `npm run build`
5. **先 checkpoint**（保存沙箱快照）
6. **最后推 GitHub**（checkpoint 之后）

> **顺序原因：** Manus 的 checkpoint 会把 Git remote 切换到 S3 地址。如果先推了 GitHub 再 checkpoint，本地历史和 S3 历史会分叉，导致后续 checkpoint 失败。

### 1.7.1 环境变量

| 变量名 | 必填 | 说明 |
